# OTT Collection Guide

이 문서는 이 저장소에서 OTT 작품 수집/검증/DB 업데이트 작업을 할 때
다음 스레드에서도 같은 흐름으로 이어가기 위한 실무 지침서입니다.

말투는 부드럽게 가도 좋지만, 판단은 꽤 보수적으로 해주세요.
목표는 "많이 넣는 것"보다 "틀리지 않게 넣는 것"입니다.

## 기본 목표

- OTT 시리즈를 수집한다.
- 플랫폼별 raw 목록을 모은 뒤, 검증을 거쳐 canonical 작품으로 저장한다.
- DB에는 `ott_canonical` + `content_platform_links` 구조로 반영한다.
- 검증에서 탈락한 stale/trash row는 link, watchlist, orphan canonical까지 같이 정리한다.
- 사용자가 결과 정리를 요청하면, 특별히 파일을 원하지 않는 한 채팅 본문에 바로 써준다.

## 기본 흐름

기본 순서는 아래를 따른다.

1. 먼저 현재 규칙으로 왜 수집/검증이 어긋나는지 확인한다.
2. 규칙을 한 군데만 고치지 말고, 같은 유형의 작품 전체에 먹히는 일반 규칙으로 바꾼다.
3. 좁은 범위 소스나 대표 작품으로 재검증한다.
4. 그 다음 풀런을 돌린다.
5. 마지막에 DB 기준 결과를 정리한다.

## 톤과 작업 방식

- 사용자는 "한두 작품만 임시로 고치는 것"보다 "규칙을 개선해서 전체 흐름이 좋아지는 것"을 선호한다.
- 그래서 예외 처리보다 일반 규칙을 먼저 생각한다.
- 다만 규칙을 넓힐 때는 잘못된 데이터를 더 많이 넣지 않도록 신뢰도 우선으로 판단한다.
- 풀런은 비싸니, 가능하면 먼저 좁게 확인하고 마지막에 한 번 돌린다.

## 현재 OTT 범위

- 기본 대상 소스:
  - `coupangplay`
  - `disney_plus`
  - `netflix`
  - `tving`
  - `wavve`
- `laftel`은 별도 취급한다.
  - 사용자가 라프텔을 포함하자고 명시하지 않는 한 OTT 정리에서는 제외해도 된다.
  - 라프텔이 범위에 들어오면 장르 수집과 무관하게 항상 `anime`로 분류한다.

## 장르 규칙

최종 장르는 아래 5개만 쓴다.

- `drama`
- `anime`
- `variety`
- `docu`
- `etc`

사용자에게 보여줄 때는 각각 이렇게 쓴다.

- `drama` -> `드라마`
- `anime` -> `애니메이션`
- `variety` -> `예능`
- `docu` -> `다큐멘터리`
- `etc` -> `기타`

분류 기준:

- `Animation`, `Anime` 계열은 `anime`
- `Documentary`, `Docu-series`, `Docu` 계열은 `docu`
- `Variety`, `Reality`, `Talk Show`, `Game Show`, `Competition`, `Survival` 계열은 `variety`
- 아래 scripted 장르는 전부 `drama`
  - `Drama`
  - `Crime`
  - `Mystery`
  - `Thriller`
  - `Romance`
  - `Legal`
  - `Medical`
  - `Fantasy`
  - `Comedy`
  - `Family`
  - `Action`
  - `Suspense`
  - `Sitcom`
- 그 외는 `etc`

주의:

- 플랫폼 페이지 내비게이션 텍스트나 마케팅 문구에서 나오는 `예능`, `드라마`, `영화`는 장르 근거로 너무 쉽게 믿지 않는다.
- 특히 쿠팡/티빙/웨이브는 페이지 본문 전체를 장르 근거로 쓰면 노이즈가 많다.
- 공식 장르 문구가 있으면 그걸 우선하고, 없으면 TMDb 장르를 장르 보강에 적극 활용한다.

## 출연진 규칙

OTT 카드의 웹툰/웹소설 `작가` 자리에는 `출연진`을 넣는다.

원칙:

- 최대 4명
- 하지만 "무조건 4명"이 아니라 "최대 4명"이다
- 신뢰할 수 있는 출연진이 2명, 3명뿐이면 그대로 둔다
- garbage text를 억지로 채우지 않는다

우선순위:

1. 공식 상세 페이지의 구조화 cast
2. 공식 상세 페이지의 `출연:` 같은 명시적 라벨
3. TMDb 구조화 cast
4. 나무위키/위키백과/기타 공개문서의 명시적 cast

하지 말 것:

- `출연한다`, `원년 멤버`, `작가`, `감독`, `프로듀서`, 설명 문장 조각을 이름처럼 뽑지 않는다
- cast가 없으면 빈 배열도 허용한다

추가 규칙:

- 공식 cast가 이미 3명 이상이면 TMDb cast를 굳이 덧붙이지 않는다
- 공식 cast가 부족할 때만 TMDb cast로 보강한다
- 한영 중복은 가능하면 줄인다. 예: `재크 브래프`와 `잭 브래프`가 동시에 들어오면 중복 가능성을 의심한다.

## 날짜 규칙

목표 필드:

- `release_start_at`
- `release_end_at`
- `release_end_status`

`release_end_status`는 아래만 쓴다.

- `unknown`
- `scheduled`
- `confirmed`

날짜 신뢰 우선순위:

1. 공식 플랫폼 상세/회차/시즌 정보
2. 나무위키/위키백과 등 공개문서
3. TMDb fallback

중요:

- TMDb는 장르/cast 보강에는 적극 활용해도, 날짜에서는 공식보다 위에 두지 않는다
- 특히 `Disney+`, `Netflix`는 TMDb가 원방일이나 이전 시즌 날짜를 주는 경우가 있어서 조심한다
- 현재 플랫폼에서 새 시즌이 올라오는 경우, 예전 시즌의 방영일을 끌고 오지 않게 주의한다

날짜 추론 규칙:

- 공식적으로 회차 수와 요일이 있고 현재 시즌 정보가 분명할 때만 완결일을 계산해도 된다
- 하지만 추론된 시즌 승격만 있고 해당 시즌의 회차/요일 근거가 없으면 완결일은 계산하지 말고 `unknown`으로 둔다

## 시즌 규칙

제목에 시즌이 안 보이더라도, 현재 플랫폼의 실제 노출 시즌을 기준으로 제목을 승격할 수 있다.

우선순위:

1. 공식 시즌 선택기/시즌 메타
2. 공식 회차 정보
3. TMDb season details / season aggregate credits
4. 나무위키/위키백과 시즌별 문서
5. 마지막으로 제한적인 휴리스틱

제목 승격 예:

- `더 피트` -> `더 피트 시즌 2`
- `인더스트리` -> `인더스트리 시즌 4`

중요:

- 시즌이 승격되면 그 제목으로 다시 검색해야 한다
- 일반 문서만 보고 멈추지 말고, `시즌 2`, `시즌 4` 같은 시즌별 문서를 한 번 더 찾는다

휴리스틱:

- 현재 수집 작품이 `연재중/예정`
- 현재 cast가 과거 cast와 확연히 다르고
- 과거 시즌 날짜가 지금 시작일보다 충분히 오래 전이면
- 다음 시즌 존재를 강하게 의심할 수 있다

하지만:

- 휴리스틱으로 시즌을 올렸더라도 그 시즌의 날짜 근거가 직접 붙지 않으면
  `release_end_at`은 추론하지 말고 `unknown`으로 둔다

## 미국/글로벌 작품 규칙

미국 드라마나 글로벌 작품은 영문명을 적극 활용한다.

사용 순서:

- 한국 제목
- 공식 페이지에 드러난 영문 alias
- TMDb `original_name`
- 위키 문서의 영문명

추가 검색:

- TMDb가 안 잡히면 `IMDb`를 검색 쿼리에 포함한다
- cast 1~2명을 함께 붙인 검색도 허용한다
  - 예: `"<제목>" "<배우1>" "<배우2>" IMDb`

## 넷플릭스 / 디즈니+ 추가 규칙

이 두 소스는 전세계 동시 공개라서 TMDb가 도움이 되는 경우가 많다.
다만 날짜는 아래처럼 쓴다.

- 공식/공개문서에 날짜가 있으면 그쪽 우선
- TMDb는 날짜 fallback
- 원방일과 플랫폼 공개일이 다를 수 있으니, 현재 플랫폼에서 새 시즌 공개 중이라면
  이전 시즌의 날짜를 그대로 가져오지 않도록 조심한다

## 쿠팡플레이 추가 규칙

- 수집 대상은 `TV프로그램, 매주 새 에피소드` 섹션
- 공식 상세 + 공식 회차 API를 적극 활용
- `rendered DOM`도 공식 근거로 활용

## 티빙 / 웨이브 추가 규칙

- API/정적 문서가 약하면 브라우저 캡처/DOM fallback 허용
- `대한민국에서 건물주 되는 법` 같은 케이스는 공식 상세 + 나무위키 시즌/방영기간을 적극 참고

## 장기 예능 규칙

장기 예능은 느슨하게 넣지 않는다.

현재 기본 원칙:

- 비시즌형 장기 예능은 제외
- 시즌형이어도 너무 오래되거나 시즌 수가 과도하면 제외

이미 사용했던 기준:

- 시즌 10 이상 제외
- 첫 방영일 10년 이상 된 작품 제외
- 시즌 구분이 불명확한 장기 예능 제외

사용자가 더 엄격히 하자고 하면 그 기준을 따른다.

## TMDb 사용 규칙

TMDb는 이미 연결되어 있다고 가정한다.

용도:

- 장르 보강
- cast 보강
- 시즌 구조 파악
- 공식/공개문서가 약한 경우 날짜 fallback

주의:

- 날짜를 과신하지 말 것
- `The Neighbors`, `Paradise` 같은 흔한 영문 제목은 동명이작이 많으니
  플랫폼 설명, cast, 시즌 구조가 맞는지 꼭 같이 본다

## IMDb 사용 규칙

IMDb는 직접 구조화 API보다 검색 fallback 성격으로 쓴다.

주로 이런 경우:

- TMDb가 제목을 못 찾을 때
- 영문 alias가 더 강할 때
- 다음 시즌 존재 여부를 확인해야 할 때

## 결과 정리 방식

사용자가 결과 정리를 요청하면 기본 형식은 아래를 따른다.

- `플랫폼 // 제목 // 방영기간 // 장르 // 출연 // url`

가능하면 최신 DB 상태 기준으로 정리한다.

## 지금까지 드러난 대표 함정

- 공식 페이지 마케팅 문구 때문에 장르가 `예능`으로 오염되는 문제
- label 없는 본문에서 cast를 뽑아 garbage text가 들어가는 문제
- 시즌 승격은 됐는데 날짜는 이전 시즌 문서를 따라가는 문제
- TMDb가 도움은 되지만, 날짜를 너무 믿으면 원방일이 섞이는 문제

## 이 문서를 읽는 다음 스레드의 기본 행동

OTT 작업이 들어오면:

1. 먼저 현재 DB 결과와 최근 run summary를 확인한다.
2. 문제가 특정 작품 하나처럼 보여도, 가능하면 일반 규칙으로 풀 수 있는지 먼저 본다.
3. 수정 후에는 좁은 소스로 재검증한다.
4. 마지막에 풀런을 돌린다.
5. 정리는 파일이 아니라 채팅 본문에 먼저 써준다. 사용자가 따로 파일을 원하면 그때 만든다.
